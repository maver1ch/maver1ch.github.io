## Problem

The legacy contract parser was a single mega-prompt over an LLM. Each layer of that simplicity had a cost.

- **Hallucinations on unfamiliar clauses** — the model invented language when it had no anchor.
- **Inconsistent outputs across runs** — the same contract returned different fields twice in a row.
- **Per-field accuracy invisible** — only aggregate scores existed; no way to pin a regression to "parties" vs "obligations."
- **Couldn't scale** — sync REST timed out as contract volume grew.
- **Unstructured output** — downstream legal systems received free text, not validated schemas.

The whole stack at a glance. The API gateway routes uploads into an **orchestrator**, which selects and runs the right Skills. Skills pull grounded context from a **Qdrant** vector store, call **Bedrock** for inference, and emit schema-validated structured output. Results land in a result store that downstream legal workflows can consume directly.

<ArchitectureSection
  step="03"
  title="Skill-Based Framework"
  image="/assets/case-studies/fpt-contract/images/3-skill-framework.png"
  alt="Skill-based extraction framework with composable skills"
  highlight
  challenge="A single mega-prompt extracting parties, dates, clauses, and obligations at once is unprompt-able. You can't tune it for one field without breaking another, and a regression is invisible until a downstream system breaks."
>
The core differentiator. Each extraction goal — **parties, effective dates, governing law, obligations, termination clauses** — is its own composable **Skill**. The orchestrator selects which Skills to run for a given contract type and composes them in sequence or in parallel. This is what unlocks the +35% accuracy: each Skill is small, focused, separately prompt-engineered, and individually evaluated. Adding a new extraction goal is a new Skill, not a rewrite.
</ArchitectureSection>

<ArchitectureSection
  step="04"
  title="Skill Structure (Anatomy)"
  image="/assets/case-studies/fpt-contract/images/4-skill-structure-anatomy.png"
  alt="Internal anatomy of a single Skill — prompt, RAG hook, schema, validators, eval hooks"
>
What lives inside one Skill. A prompt template tuned to a single extraction goal. A **RAG retrieval hook** that pulls relevant reference clauses from Qdrant. An **output schema** with Pydantic-style validation so downstream systems never receive garbage. Pre- and post-call **validators** and **eval hooks** that score the extraction in production. Every Skill is independently testable and replaceable.
</ArchitectureSection>

<ArchitectureSection
  step="05"
  title="RAG Pipeline"
  image="/assets/case-studies/fpt-contract/images/5-rag-pipeline.png"
  alt="Qdrant-backed RAG pipeline for contract clause grounding"
>
The retrieval side. Contract language varies by jurisdiction and template; a raw LLM call hallucinates on edge cases. **Qdrant** stores embeddings of a reference clause corpus, and each Skill retrieves the most relevant clauses before calling Bedrock. Grounding the LLM in known patterns is what brings hallucination rates down on unfamiliar language.
</ArchitectureSection>

<ArchitectureSection
  step="06"
  title="AWS Infrastructure"
  image="/assets/case-studies/fpt-contract/images/6-aws-infrastructure.png"
  alt="AWS production infrastructure: S3, Lambda, SQS, Bedrock, Qdrant"
  highlight
>
Production deploy is fully AWS-native. Contracts land in **S3**, which triggers **Lambda** to enqueue extraction jobs in **SQS**. Workers consume the queue, run Skills against **Qdrant** and **Bedrock**, and write structured results back to S3 and the result store. VPC and IAM hardening keep the data plane locked down. No vendor lock-in beyond AWS itself.
</ArchitectureSection>

<ArchitectureSection
  step="07"
  title="Async Batch Jobs"
  image="/assets/case-studies/fpt-contract/images/7-async-job-batch.png"

Sync REST cannot stream thousands of contracts through extraction. The async batch path uses **SQS-driven workers with idempotent handlers and retries**, so a failing contract doesn't poison the batch and a worker restart doesn't double-charge Bedrock. Progress is reported back to the user as the batch advances, not just at completion.
</ArchitectureSection>

## Outcome

- **+35% extraction accuracy** vs the legacy single-prompt system.
- Scales to **thousands of contracts per batch** via SQS + async workers.
- **Skill-based decomposition** unlocked per-task evaluation and surgical regression fixes.
- **RAG-grounded extraction** cut hallucination rates on unfamiliar contract language.
- **AWS-native deploy** on S3 + Lambda + SQS + Bedrock + Qdrant — no vendor lock-in beyond AWS.
- The framework is reusable for adjacent legal workflows: NDA review, clause comparison, obligation tracking.
